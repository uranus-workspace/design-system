# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

This is the design system monorepo for **Uranus Technologies** (https://uranus.com.br/). It produces:

- **`@uranus-workspace/design-system`** — publishable core primitives (shadcn/ui sources + Tailwind v4 + Motion).
- **`@uranus-workspace/blocks`** — publishable composite components built on top of the core package.
- **`@uranus-workspace/ai`** — publishable AI surfaces (chat, composer, streaming, reasoning) built on the design-system package and the Vercel AI SDK.
- **`@uranus-workspace/auth`** — publishable OIDC authentication package (oidc-client-ts + Keycloak presets, React provider/hooks/guards, pre-wired screens, optional Next.js JWKS middleware).
- **`apps/docs`** — public docs site (Next.js 15 + Fumadocs MDX).
- **`apps/storybook`** — Storybook 8 component browser for all packages.

The Uranus branding manual PDF is the source of truth for visual identity. Placeholder tokens ship in `packages/tokens/src/*.ts` until the manual is ingested.

## Stack

- **pnpm 10.33.0** workspaces + **Turborepo 2** (Node 22)
- **Biome** (no ESLint, no Prettier — do not add them)
- **TypeScript 5.7**, strict, `noUncheckedIndexedAccess`
- **Tailwind CSS v4** with CSS-first `@theme`, tokens via CSS custom properties
- **Motion** (successor to Framer Motion) — peer dep on both published packages
- **Vitest + React Testing Library + jest-axe** for unit + a11y tests
- **Storybook 8** with Vite builder, `addon-a11y` set to `test: 'error'`
- **Fumadocs MDX** (`fumadocs-ui` + `fumadocs-mdx`) for the docs site
- **tsup** for library builds (ESM + CJS + `.d.ts`, `"use client"` banner injected)
- **Changesets** for versioning and publishing

## Architectural invariants

Preserve these when making changes — they are the reason the repo is set up this way:

1. **Tokens flow one direction.** Branding manual → `packages/tokens/src/*.ts` → `packages/tokens/dist/tokens.css` (generated by `scripts/build-css.ts`) → `packages/tailwind-config/preset.css` → every app and package. Never hardcode hex, rgb, or px values in components. Never reference raw ramp steps (`--color-brand-600`) in component code — use semantic tokens (`--color-primary`).
2. **Docs and Storybook import the real library.** Live examples in `apps/docs/content/docs/**/*.mdx` and stories in `packages/*/src/**/*.stories.tsx` import from `@uranus-workspace/design-system` / `@uranus-workspace/blocks` via workspace deps. Never duplicate a component just to document it.
3. **shadcn components are owned source, not a dependency.** Use `pnpm dlx shadcn@latest add <name>` from inside `packages/design-system` — the CLI is pre-wired via `components.json` to write into `src/components/<name>/` using `@uranus-workspace/tokens`. Customize the generated code; don't lock it down.
4. **Biome replaces ESLint + Prettier.** The root `biome.json` extends `@uranus-workspace/biome-config`. Do not add `.eslintrc*`, `.prettierrc*`, or their plugins.
5. **Per-component contract.** Every component in `design-system` or `blocks` ships with five artifacts in a single folder: implementation, cva variants file (when applicable), `*.test.tsx` (Vitest + RTL + jest-axe), `*.stories.tsx` (Storybook), and an MDX page under `apps/docs/content/docs/{components,blocks}/`. No exceptions.
6. **`a11y: test: 'error'` is set on every Story's parameters.** New components must pass Storybook's a11y addon as errors, not warnings. `jest-axe` calls in unit tests must return `toHaveNoViolations()`.
7. **Blocks compose primitives, never reinvent them.** `@uranus-workspace/blocks` depends on `@uranus-workspace/design-system` via `workspace:*` peer. A block that reimplements a button is a bug.

## Blocks authoring (composition-first)

When adding or refactoring `@uranus-workspace/blocks` components:

- **Prefer `children` and named slots (`ReactNode`)** over serializable config trees when structure is app-specific or will vary (navigation, toolbars, custom rows).
- **Use compound components** (`Parent`, `Parent.Slot`) when you need clear placement without one undifferentiated `children` blob.
- **Use render props sparingly** — only when parent state must flow into the child (`renderItem`, etc.); keep signatures narrow.
- **Compose other blocks or primitives** instead of duplicating dialog/sheet/list wrappers; share a11y and Radix wiring in one place.
- **Auth flows** — `SignInForm`, `SignUpForm`, `ForgotPasswordForm`, `ResetPasswordForm`, and `OtpVerificationForm` expose colocated Zod schemas (`*.schema.ts`) and rely on `react-hook-form` wired to `Form` primitives from the design-system package. Use `OAuthProviderButton` + `AuthDivider` for SSO + email composition. Peers: `react-hook-form`, `zod`, `@hookform/resolvers` (see `packages/blocks/package.json`).

See [`packages/blocks/README.md`](packages/blocks/README.md) for a short package-level summary.

## Commands

From the repo root:

```bash
pnpm install                 # install all workspaces
pnpm dev                     # turbo run dev (all apps + lib watchers)
pnpm build                   # turbo run build across everything
pnpm lint                    # Biome check
pnpm lint:fix                # Biome check --write
pnpm typecheck               # tsc --build across references via turbo
pnpm test                    # Vitest across both packages
pnpm test:watch              # Vitest in watch mode
pnpm storybook               # Storybook dev server (port 6006)
pnpm build-storybook         # Static Storybook build
pnpm docs:dev                # Fumadocs dev server
pnpm docs:build              # Static docs build
pnpm changeset               # create a changeset for a release
pnpm release                 # build + changeset publish (used by CI)
```

Scope to a single package:

```bash
pnpm --filter @uranus-workspace/design-system <script>
pnpm --filter @uranus-workspace/blocks build
pnpm --filter @uranus-workspace/docs dev
```

Run a single Vitest file:

```bash
pnpm --filter @uranus-workspace/design-system test -- src/components/button/button.test.tsx
```

Add a shadcn component:

```bash
cd packages/design-system
pnpm dlx shadcn@latest add dialog
```

## Authoring a new component (the golden path)

When asked to add a component (primitive or block), the expected flow is:

1. **Primitive?** `cd packages/design-system && pnpm dlx shadcn@latest add <name>`. Then move the generated files into a `src/components/<name>/` folder.
2. **Block?** Create `packages/blocks/src/components/<name>/` by hand. Import primitives from `@uranus-workspace/design-system`.
3. Create the five required files: `<name>.tsx`, `<name>.variants.ts` (if variants exist), `<name>.test.tsx`, `<name>.stories.tsx`, `index.ts`.
4. Export the component from the package barrel (`src/index.ts`).
5. Create the MDX doc page at `apps/docs/content/docs/{components,blocks}/<name>.mdx` using the existing pages as templates — anatomy, variants, usage, do/don't, accessibility.
6. Add the page slug to the relevant `meta.json` in `apps/docs/content/docs/`.
7. Run `pnpm lint && pnpm typecheck && pnpm test` and `pnpm changeset` before opening the PR.

## Key files to know

- `packages/tokens/src/index.ts` — every design token starts here. Changing a value here propagates to CSS via the generated `dist/tokens.css`.
- `packages/tailwind-config/preset.css` — the Tailwind v4 `@theme` block mapping tokens to semantic names (`--color-primary`, `--color-ring`, etc.).
- `packages/design-system/components.json` — shadcn CLI config. Already wired for the monorepo.
- `packages/design-system/src/lib/cn.ts` — the one `cn()` helper. Do not duplicate it.
- `packages/design-system/src/lib/motion.ts` — shared Motion variant presets. New animations should reuse these.
- `packages/design-system/tsup.config.ts` — library build config. Note the `"use client"` banner injected on every entry.
- `packages/test-config/setup.ts` — jest-dom, cleanup, jest-axe extension, matchMedia + ResizeObserver mocks for jsdom.
- `apps/docs/content/docs/components/button.mdx` — canonical template for a component MDX page.
- `apps/storybook/.storybook/main.ts` — glob-based story discovery across both packages.

## What is intentionally missing

- **Most shadcn primitives.** Only `Button`, `Input`, `Card` ship in the initial scaffold. Add the rest via the shadcn CLI as they are needed — do not rewrite the scaffold to preinclude them.
- **Most blocks.** Only `PageHeader` and `EmptyState` ship initially. Planned follow-ups live in `apps/docs/content/docs/resources.mdx` under Roadmap.
- **Real branding.** Tokens are placeholder OKLCH values with structure stable. When the Uranus branding manual PDF is provided, edits are local to `packages/tokens/src/*.ts`.
- **Deployment targets.** CI builds everything; Vercel/npm tokens still need to be added to GitHub secrets before the first release.
- **Chromatic / visual regression.** Deliberately out of scope for the scaffold.
